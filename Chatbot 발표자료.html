1. 개인 목표 설정

제안서의 리스크를 확인하고 점수를 제공하는 RAG 서비스

1.1. RAG(Retrieval-Augmented Generation) 에 대한 이해

LLM이 답을 만들어 내기 전에 답을 만들기 위한 재료를 먼저 찾아서 준다.
RAG는 저장(인덱싱) 과 질의(검색)으로 구성됨, 두 단계 모두 Embedding이 필요하다.
Fine-tuning의 경우 데이터 변경시 모델의 재 학습이 필요하기에 발생 비용이 큼, RAG는 DB만 갱신하면 된다.
개념적 파이프라인: 사용자 질의 > Query Embedding > VectorDB 검색 > 관련 문서 chunk 추출 > prompt Contenxt 주입 > LLM 생성 > 답변

1.2. 필요한 기술의 특징 이해 및 적용

Embedding:
개념: 비정형 데이터를 고정 길이의 수치 백터로 변환하는 기술이다. 이 벡터는 의미적으로 유사한 데이터 끼리 공간적으로 가깝게 배치되도록 학습되어 있다. (의미를 좌표에 투영함)
LLM은 긴 문서를 그대로 기억할 수없고, 수천 수만의 문서를 직접 비교할수 없다. Embedding을 사용하면 의미기반으로 비교가 가능하고 키워드가 달라도 의미가 같으면 검색이 가능하다.
Chunk 단위로 Embedding시 Chunk의 크기가 너무 크면 의미가 희석되고, 작으면 문맥이 손실된다.
제안서/문서 검색에는 bge-m3, E5 계열, multilingual 계열 추천
LangChain:
개념: LLM 기반 애플리케이션을 구성하기 위한 오케스트레이션 프레임워크이다. Prompt, Retriever, Chain, Tool, Memory를 파이프라인 형태로 조합할 수 있게 해준다.
RAG 시스템은 질문 전처리, 검색, 문서 가공, 프롬프트 구성, LLM 호출, 출력 파싱, 스트리밍 등이 필요하고 해당 기능들을 각 각 구현시 유지보수 및 흐름파악, 확장에 취약하다. LangChain은 이를 Runnable Graph로 구조화한다.
RAG의 복잡한 흐름을 눈에 보이게 만드는것이 장점
VectorDB:
개념: 고차원 백터를 저장하고, 유사도 기반 검색을 수행하는 데이터베이스이다.
저장되는 Embedding은 단순 숫자 배열, 수만~수백만 백터를 밀리초 단위로 Top-k 유사도 검색 가능
메타데이터를 통한 필터링 가능

pdf 텍스트 추출시 생각보다 텍스트가 아닌 이미지 내부 글이 많고, 문장으로 끝나는게 아닌 특성별 기술 형태의 글이 많음, 노이즈 제거 필요

2.2. 문서 내 그림 및 표 처리 방안

문서 내 그림 및 표에 주요 정보들이 많음, 해당 정보들을 ocr를 통해 텍스트 변환 후 제공을 하거나 의미 해석이 필요함
그림 및 표에 대한 분석을 못할 경우 문서 전체를 알고있다고 할 수 없음
최소한 근거 자료로 제공이 되야함

2.3. Embedding 처리 방안

같은 목차일 경우 대부분 큰 맥락을 유지한다고 볼 수 있음, 임베딩 시 그 맥락의 관점에서 의미기반으로 chunk 필요
이미지 임베딩 검토 결과 이미지 자체를 임베딩하게되면 이미지 끼리의 유사도 비교가 가능하지 이미지 내 의미 파악은 불가능, summary 필요
표 역시 문장이 아니기에 노이즈로 오해될 수 있을것 같음(짐작) > 어떤 표 인지 caption이나 summary가 필요할 것으로 고려됨

6. Embedding

Embedding의 전반적인 절차
Raw Text > 1.Tokenizer > 2.Token Embedding Layer (ID > Vector) > 3. Transformer Encoder > 4. Pooling > 5. Sentence Embedding > 6. Normalization > 7. VectorDB 저장/검색
Tokenizer: 텍스트를 의미 기반 최소 단위(토큰)으로 분해 하고 토큰의 아이디를 반환함
"임베딩 테스트" → ["임", "##베", "##딩", "테스트"] → [1032, 9812, 554, 3201]
Token Embedding Layer: 각 토큰의 ID를 고정차원의 백터로 변환
1032 → [0.12, -0.88, 0.44, ...] # hidden_dim
Transformer Encoder: self-attention을 통해 토큰 간 관계를 계산, 문맥을 반영한 백터 생성
입력: [임] [##베] [##딩] [테스트] 출력: 각 토큰마다 contextual embedding
부연설명. 예제 ‘매출’이 수치인지 목표인지 문맥구분, ‘상반기’ 가 일정 의미임을 파악
출력 형태: [sequence_length, hidden_dim]
Pooling: 문장 단위로의 압축
Sentence Embedding: 최종 백터이며 문장/문단/청크의 의미를 대표함
[hidden_dim] # 예: 768, 1024, 4096 (Pooling 결과)
Normalization(선택): Cosine Similarity 계산 및 안정화, 길이차이 제거, VectorDB 점수 일관성 확보
||v|| = 1 - 백터는 방향(의미, direction)과 길이(값의 크기, magnitude)를 가진다, Normalization을 통해 방향은 유지하면서 길이 값을 1로 변경 함으로써 크기차이를 제거함
- “의미는 덜 비슷한데, 벡터가 길어서 점수가 더 높음” >> 문제를 해결함
Embedding의 방법
Dense Embedding
개념: 텍스트 전체의 의미를 하나의 실수 벡터로 압축 한것, 의미 후보 생성기
Sparse Embedding
개념: 키워드 기반
종류: Lexical(BM25, TF-IDF), Neural(SPLADE 계열)
Hybrid Embedding
개념: Dense + Sparse 를 통해 검색 점수를 결합하는 방식, 의미기반 검색(Dense)의 장점과 키워드 기반 검색(Sparse)의 정확도를 동시에 확보하기 위한 접근법
Re-Rank: Hybird 방식은 그럴듯한 문서들의 집합이므로, 질문에 가장 정확히 답하는 문서의 순위는 아니다.
(final_score = α × dense_score + β × sparse_score)
Query와 각 문서의 쌍을 통해 문서 후보군에서 의도를 파악하기 위해 사용한다. (Hybrid가 뽑은 후보군을 정답에 가까운 순서로 재정렬)

9.2. 로컬 임베딩

SentenceTransformer가 내부적으로 하는 일을 OpenAI API가 서버에서 대신 해주는 구조 openai embedding 모델을 사용할떄와 다르게 아래 처리 필요 • tokenizer • encoder • pooling • normalize [ DENSE ] tokenizer → encoder(last_hidden_state) → mean pooling (attention mask) → L2 normalize → dense vector SentenceTransformer, FlagEmbedding는 이런 상황에서 필요: • 로컬 모델 • HuggingFace 모델 • PyTorch 직접 로딩 bge-m3를 쓰면 • dimension: 1024 • SentenceTransformer or FlagEmbedding 필요 • 로컬 / MPS / CPU 가능 • VectorDB 비용↓, 속도↑ OpenAI 임베딩을 쓰면 • dimension: 1536 or 3072 • SentenceTransformer • API 호출 필수 • 검색 시에도 같은 OpenAI 모델을 계속 사용해야 함
